# 10.1 - What Is an AI Agent?

## The definition

An **agent** is an LLM + **tools** + a **loop**. The LLM decides what to do, runs a tool, observes the result, decides what to do next. Repeat until done.

The simplest one:

```
while not done:
    thought = llm.generate("What should I do given the history?")
    tool_name, tool_args = parse(thought)
    observation = run_tool(tool_name, tool_args)
    history.append((thought, observation))
```

That loop is the skeleton of every agent system.

## The spectrum

Not all "agentic" things are equally autonomous:

### Level 0: single-turn chat
You ask ChatGPT a question, it answers. No tools, no loop, no state.

### Level 1: single tool call
LLM calls one external function, returns result. Like: "What's 237 × 491?" → model calls calculator → "116,367".

### Level 2: multi-tool chains
LLM plans a sequence of tool calls. "Research the best restaurants in Seattle and put them in a table" → Search → Browse → Extract → Format.

### Level 3: self-correcting agent
LLM tries something, sees it fail, tries something else. Can recover from errors. Example: code that doesn't compile → reads error → fixes.

### Level 4: long-horizon autonomous agent
Given a high-level goal, plans over many steps, persists state, makes progress over hours or days. Still mostly aspirational in 2026.

Most "agents" in production today are Level 1-2. Level 3 is emerging (Cursor, Devin). Level 4 is research.

## When do you need an agent?

You need agents when:
- The task requires **external information** (search, database, API).
- The task requires **actions** (send email, book flight, execute code).
- The task has **multiple steps** that depend on intermediate results.

You don't need agents when:
- A single LLM call with good context suffices.
- Adding tool calls just adds latency and failure modes.

Don't agent-ify unnecessarily. Most "agentic products" would be better as better-prompted LLM apps.

## Components of an agent

### 1. The LLM (the "brain")
GPT-4, Claude, Gemini, Llama - something capable enough to:
- Understand task intent.
- Reason about tool choice.
- Parse tool responses.
- Decide "done" correctly.

Models trained specifically for tool use (Claude 3.5 Sonnet, GPT-4o, Llama 3.1+) work much better than generic LLMs.

### 2. The tools
Functions the LLM can call. Each has:
- A **name** (e.g., `search_web`).
- A **description** (when to use it).
- A **schema** (input parameters).
- An **implementation** (the actual Python function).

Tools can be: search, calculator, code execution, database queries, file I/O, API calls.

### 3. The loop
The harness that:
- Sends messages to the LLM.
- Parses tool calls from the response.
- Executes tools.
- Sends observations back to the LLM.
- Decides termination.

### 4. Memory (optional)
- **Short-term**: the conversation history.
- **Long-term**: vector store of past interactions, summary of episodes, user profile, etc.

### 5. Guardrails (production)
- Allowlist of tools.
- Confirmation for dangerous actions.
- Rate limits.
- Cost caps.

## The ReAct pattern

**ReAct = Reasoning + Acting** (Yao 2022). The standard pattern:

```
Thought: I need to find the weather in Paris.
Action: search_web("Paris weather today")
Observation: Partly cloudy, 18°C, 62% humidity.
Thought: Now I have the answer.
Final Answer: The weather in Paris is partly cloudy and 18°C.
```

The "Thought" and "Final Answer" are generated by the LLM. "Action" is parsed and executed by your harness. "Observation" is the result inserted back into the context.

Almost all modern agent systems are variants of ReAct.

## Example: a tiny agent

```python
import openai
from duckduckgo_search import DDGS

def search_web(query):
    with DDGS() as ddgs:
        results = list(ddgs.text(query, max_results=3))
    return "\n".join(r["body"] for r in results)

tools = [{
    "type": "function",
    "function": {
        "name": "search_web",
        "description": "Search the web for up-to-date info.",
        "parameters": {
            "type": "object",
            "properties": {"query": {"type": "string"}},
            "required": ["query"]
        }
    }
}]

messages = [{"role": "user", "content": "What's happening in AI this week?"}]

while True:
    response = openai.chat.completions.create(
        model="gpt-4o-mini", messages=messages, tools=tools
    )
    msg = response.choices[0].message
    messages.append(msg)
    
    if not msg.tool_calls:
        print(msg.content)
        break
    
    for tc in msg.tool_calls:
        result = search_web(**json.loads(tc.function.arguments))
        messages.append({
            "role": "tool", "tool_call_id": tc.id, "content": result
        })
```

~30 lines. That's a working agent. From here you add: more tools, memory, guardrails, UIs.

## Why now?

Why is agentic AI suddenly huge?

1. **LLMs got smart enough** to reliably pick tools and parse JSON schemas.
2. **Function calling APIs** (OpenAI 2023, Anthropic 2024) standardized the interface.
3. **Long context** (128K+) lets agents maintain state.
4. **Inference became cheap enough** that running 10-50 LLM calls per task is viable.
5. **Reasoning models** (o1, R1) plan multi-step actions much better.

The pieces only came together in late 2023. We're still in the early days.

  Level 4 still aspirational.

**The tiny-agent in 30 lines**:

```python
import json
from openai import OpenAI

client = OpenAI()

def search_web(query):
    # imagine this uses DDG or Google
    return f"[search results for: {query}]"

tools = [{
    "type": "function",
    "function": {
        "name": "search_web",
        "description": "Search the web for current info.",
        "parameters": {
            "type": "object",
            "properties": {"query": {"type": "string"}},
            "required": ["query"],
        },
    }
}]

messages = [{"role": "user", "content": "What's AI news this week?"}]

while True:
    r = client.chat.completions.create(
        model="gpt-4o-mini", messages=messages, tools=tools,
    )
    msg = r.choices[0].message
    messages.append(msg.model_dump())

    if not msg.tool_calls:
        print(msg.content)
        break

    for tc in msg.tool_calls:
        args = json.loads(tc.function.arguments)
        result = search_web(args["query"])
        messages.append({
            "role": "tool", "tool_call_id": tc.id, "content": result
        })
```

**Why now? (a timeline)**:

```
  Capability gate                        Opened by
  ────────────────────                   ────────────────────
  LLMs smart enough to call tools        GPT-3.5 / Claude 1  (late 2022)
  Function calling APIs standardized     OpenAI (June 2023)
  Context long enough (100K+)             GPT-4-Turbo (late 2023)
  Reasoning models emerged                o1, R1 (late 2024)
  Cross-provider tool standard             MCP (Anthropic, Nov 2024)

  → Agentic AI became practical ~2023.
  → Became mainstream ~2024.
  → We're in the "useful but unreliable" era now (2026).
```

**When to agent vs when not**:

```
  USE agent patterns when:
    ✓ Task requires EXTERNAL info (search, database, API)
    ✓ Task requires ACTIONS (send email, write file)
    ✓ Task has MULTIPLE STEPS that depend on each other
    ✓ Task requires REASONING about tool results

  DON'T use when:
    ✗ Simple question with 1-shot answer
    ✗ Retrieval could solve it (just use RAG)
    ✗ Fixed workflow (use explicit code, not LLM)
    ✗ Low reliability acceptable (agents add failure modes)
```

**Agent building blocks**:

```
  To build an agent, you need:

  1. LLM API or local model
  2. Tool definitions (JSON Schema)
  3. Tool implementations (Python functions)
  4. Loop runner (~30 lines)
  5. Error handling (tool fails, LLM outputs bad JSON)
  6. Logging (trace every action for debugging)
  7. Budget guards ($ cap, step cap, time cap)
  8. (Optional) Memory
  9. (Optional) Safety layer (confirmation for dangerous actions)

  You can start with just 1-5. Add 6-9 as you scale.
```

**Common agent types**:

```
  Research agent:          search → summarize → answer
  Code agent:              read → modify → run → test → iterate
  Data analyst:            upload CSV → explore → visualize → explain
  Customer support:        RAG over docs → answer with citations
  Web scraper:             browse → extract → output structured data
  Scheduler:                understand → check calendar → book
```

Each is a different combination of LLM + tools. Core pattern is the same.

**Real agents shipping in 2026**:

```
  Consumer-facing:
    - Cursor (code editor with AI agents)
    - Claude Code (Anthropic's code agent)
    - ChatGPT code interpreter (data analyst)
    - GitHub Copilot Workspace

  Research:
    - Devin (Cognition): autonomous SWE agent
    - AutoGPT, BabyAGI (early demos, now superseded)
    - LangChain + OpenAI Assistants API

  Enterprise:
    - custom RAG-chatbots over company docs
    - customer support automation
    - business intelligence agents (text-to-SQL)
```

**The reliability challenge**:

```
  Today's agents:
    ✓ Simple, narrow tasks: 90% reliable
    ✓ Multi-step in familiar domains: 60-80% reliable
    ✗ Novel long-horizon tasks: 20-40% reliable

  Unreliability sources:
    - LLM confabulation (invented facts)
    - Tool errors (API down, bad JSON output)
    - Loop getting stuck (same action repeated)
    - Context overflow (forgets earlier steps)
    - Misinterpreted instructions

  → Real agents need monitoring, retries, human escalation.
```
